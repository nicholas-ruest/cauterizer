# Abuse-Case Test Matrix

Status: P00 executable security specification  
Governing decisions: ADR-001, ADR-003 through ADR-007, ADR-010 through ADR-016, ADR-018 through ADR-020, ADR-022, and ADR-024  
DDD contexts: all contexts, with direct focus on Organization & Access, Remediation Runs, Isolated Execution, Patch Proposals, Verification, Evidence, and External Actions

## Test contract

Each case below is a falsifiable architectural claim. `Automated` cases run in CI or a backend qualification environment. `Procedural` cases produce signed review or drill evidence. A conformant release records the implementation version, policy/schema versions, sandbox backend and image identity, test seed, artifact digests, result, and any approved exception.

Critical cases are fail-closed and cannot be quarantined or silently retried to green. A local weak sandbox may exercise the test harness, but passing it does not satisfy hosted containment cases. All tests use synthetic secrets and fixtures; no test intentionally targets a live third-party system.

## Minimum release gates

1. Tenant isolation: AC-001 through AC-003 pass against API, database predicates, object storage, events, caches, logs, and job capabilities.
2. Sandbox isolation: AC-004 through AC-009 and AC-030 pass on the exact hosted backend/runtime/image/kernel combination. Results from a weaker local backend are informative only.
3. Solver/verifier separation: AC-011 through AC-015 pass, and the topology/data-flow review records that no undeclared channel exists.
4. Deterministic verification and evidence integrity: AC-016 through AC-020 pass with generated negative cases and offline verification.
5. Human-gated export: AC-022 through AC-024 pass; no external mutation capability is present in the MVP binary, policy, or adapter registry.
6. Cross-cutting production controls: AC-021 and AC-025 through AC-032 pass or, for explicitly procedural cases, have current signed drill evidence.

Any critical failure blocks the conformant release. An exception must identify the failed claim, affected deployment/profile, owner, compensating control, expiry, and approval by the accountable security decider; it must also force truthful non-conformant labeling wherever the failed claim is material.

## Alert and audit linkage

Production-capable implementations attach alert identifiers to repeated cross-organization denials, capability misuse, sandbox egress attempts, cleanup failures, verifier-store access by a solver identity, signer-policy denial spikes, secret-redaction detections, break-glass use, event authentication failures, and export authorization failures. Alert payloads contain opaque organization/run references and reason codes, never secret or verifier-restricted values. Tests assert both the security outcome and the presence of the expected redacted audit/alert signal.
